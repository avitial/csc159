CpE/CSc 159 Operating System Pragmatics -- Syllabus

Course Description

Application of operating system principles to the design and implementation of a multitasking operating system. You will write an operating system for a computer platform. Topics include: scheduling of processes, control and allocation of computer resources, and user interfacing. 3 units. Prerequisite: CSc 139.

Course Objectives

  • Develop thorough understanding of the design and implementation of a operating system (the kernel level, mostly)
  • Provide you with practical experience of coding a working OS kernel and other OS components
  • Present software development techniques: source code control, team work, effective use of development tools

    • Course Outcomes

  • You will follow a design guideline and write code to build an OS kernel and associated system utilities.
  • You will learn to develop large-scale software in a team environment and apply software engineering skills to manage source code development, conduct testing, and schedule regression.
  • You will be able to relate security and vulnerability issues into designs and implement solutions.
  • You will learn tools for embedded-system development.

    Course Overview

    Modern operating systems (OS) allow CPUs and other resources to be shared among many users (and processes) concurrently. The OS provides resource management and controls among processes to guarantee the correctness of operation. This course is designed to give computer science and computer engineering students an in-depth exposure to the OS operation through extensive implementation of the OS structures.

    This course is intended to provide you with practical, hands-on experience in the details of OS design and implementation. The various OS principles covered in CSc139 are considered from the viewpoint of actual code implementation on computers. After spending covering the architectural details of the target machine, you will start to write interrupt-handler code for various type of interrupts. Special emphasis will be given to the implementation of thread process creation, dispatching code, synchronization, and to the development of a complete set of OS kernel services capable of supporting typical embedded-system applications. You will implement several client/server-type processes running on the target machine using inter-process communication (IPC) mechanisms provided by their kernels. Kernel threads will provide services to the upper layers of the system.

    Attention will be given to the development and debugging techniques typical in embedded/remote-target computer systems as well as stand-alone general-purpose machines. You will download and debug OS routines from a separate workstation.

    Emphasis will be placed on the design, implementation techniques, and tradeoffs typically in the development of modern operating systems, from the viewpoint of both general-purpose time-sharing systems and ubiquitous embedded systems. Discussion will include essentials and advanced issues in the building of real-time systems, embedded systems, and multiprocessor systems.

    Operating systems are complex, and, by their nature, are very sensitive to the orders of events. Writing code is a form of communication both to yourself and other programmers, as well as to the actual CPU that executes the compiled code. You are expected to demonstrate organization and documentation skills in their programming assignments. which is important since each assignment builds on the previous ones. Comments serve to communicate critical issues and detail of possible hazards in the code.

    After completing this course, you will have gained a comprehensive understanding of the organizational structure of modern operating-system code. They will have achieved considerable practical experience with implementation methods and tools similar to those used in computer systems development in the industry.

    Course Grading

    The semester letter grade will be based on the distribution curve, derived from the total points earned from all project assignments (50%) and written exams (50%) -- midterm and final exams.

    This class requires a lot of time from you. The program you create for this class will probably be the largest computer program you've written here at school. Additionally, each assignment is the starting point for the next assignment, which is the starting point for even the latter assignment, etc. So, do not get behind in the assignments.

    Programming assignments must be turned in electronicly (will be tested and asked for demonstration). All late assignments are subject to a fixed 10% reduction each day (time stamps are given by the source code control system on the server host. 10% means 1 point if 10 points based.)

    Please inform yourself of the Department-College-University policies on dropping courses. An "incomplete" grade can only be given due to unusual circumstances caused by reasons beyond your own control. This normally means a letter from your medical doctor or employer, on appropriate letterhead, stating that you are unable to complete the semester attendances and assignments due to illness or workload.

    Project Grading Policy

    The programming projects will be graded with the following policy.
    Grading is based on a 10-point scale
    • 5 points - meeting functional requirements
    • 5 points - correct source code (functional requirements first):
      • 0.5 point, a minor error (compiler warnings, repeated initializations, etc.)
      • 1 point, a major error (array index may go over, register names swapped, etc.)
    • The typical 159 problems:
      • fail to generate target OS during compile
      • fail to start runtime
      • fail to sustain runtime
      • fail to enter the debug mode as required
      • fail to proceed to exit as required
      • runtime exceptions/errors such as segfaults, SPEDE panics
      • runtime causes reboots
      • incorrect algorithms (may likely cause crash or OS malfunction)
      • OS features or data structures implemented incorrectly
      • incorrect semantic logic in code modularization
      • certain runtime requirements not met
      • (Surely many little things may also go wrong. Best re-study, re-check.)

    Cheating and plagiarism are dealt with strictly in this class. All assignments require your own work during the current semester. It is OK to discuss assignments with others for the purpose of understanding the requirements but never about the coding. All work on team assignments for the class is to be done solely by yourself or your team member unless explicitly stated otherwise. This includes problem analysis, solution design, coding and testing. When you or your team submits work, it constitutes a contractual agreement that the work is solely the product of the named person or team members. Further, submitted work carries with it an agreement that the instructors can quiz you on the detail of the work for verification.

    If anyone attempts to raise their grade through unethical means such as working with or copying the work of others, it is in essence may cause a lowering of others' grades. Further, anyone who gives such help is equally guilty of unethical behavior because they failed to protect the proprietariness of the information.

    The 159 policy on ethics is succinctly stated in the University's Policy Manual: Cheating is the ground for dismissal from the University.
